## CRM Stabilization — 2026-03-28 Late Session

### What happened
CRM UI had multiple simultaneous crashes and data-shape contract mismatches across 7 pages. Coordinated pass: Scout (audit) → Forge (fix) → Nero (verification).

### Root cause
DB stores integers (priority 1–4, `follow_ups` snake_case), frontend utilities assumed string labels and camelCase. Multiple independent locations hit the same pattern.

### Fixes applied (CRM/ui/src/)
- `utils.js` — added `normalizePriority()` mapping int 1→critical, 2→high, 3→normal, 4→low; both `priorityVariant()` and `priorityLabel()` use it
- `pages/FollowUps.jsx` — `data?.followUps` → `data?.items`; `item.priority.toUpperCase()` → `priorityLabel(item.priority)`
- `pages/Contacts.jsx` — `data?.total` → `data?.count ?? data?.items?.length`
- `pages/Dashboard.jsx` — `weeklyStats.totalContacts` → `weeklyStats.contacts_created`; also moved `useState` before early returns (Rules of Hooks fix)
- `pages/ContactDossier.jsx` — `follow_ups: followUps` (snake_case rename); removed nonexistent `drafts` destructure and prop from `DraftsTab`
- `pages/Settings.jsx` — moved all `useState` before early returns; data-derived state synced via `useEffect`
- `components/NLQueryBar.jsx` — `results.results` → `results` (API returns array at top level); `item.score` → `item.relationship_score`

### Broader pattern found
4 distinct failure categories: integer-as-string crashes, wrong response key names, snake_case vs camelCase mismatches, nonexistent API fields.
