## Additional Entries — 03:15 to 03:54

### Subagent Delivery Gap Bug Hunt

**Root causes found:**

1. **Delivery queue clog (primary)** — Telegram heartbeat messages were stuck in `~/.openclaw/delivery-queue/` trying to reach non-existent `@heartbeat` recipient. This silently blocked ALL subagent announce messages. Fixed by clearing queue + gateway restart.

2. **Transcript context gap (secondary)** — When Sentinel runs exec and needs to synthesize a review from the output, exec results don't always propagate to the LLM's context in the next turn. The agent had nothing to synthesize from. Not a delivery bug — an agent context routing quirk.

**Test matrix:**
- Trivial text subagents (main/sentinel): ✅ delivered cleanly
- Sentinel with exec + short reply: ✅ delivered cleanly  
- Forge structured review: ✅ delivered cleanly
- Sentinel with exec + synthesis from results: ⚠️ empty (exec output didn't reach context)

**Fixes applied:**
- Cleared delivery queue: `rm -f ~/.openclaw/delivery-queue/*.json`
- Restarted gateway
- Created `SUBAGENT_DELIVERY_GAP_BUG_HUNT.md`

**Sentinel delivery gap is NOT general** — only triggered by multi-step exec+synthesis tasks. The mechanism is sound.

### Heartbeat Telegram Queue Fix

**Root cause:** Main agent heartbeat was configured `deliver.channel: "telegram"` — but Telegram channel had no valid `announceTo` recipient. Heartbeat tried to announce to `@heartbeat` (doesn't exist), failed, got stuck in queue. Queue clogged with 11+ failed items, silently blocking ALL subagent announces.

**Fix applied:**
- Changed main agent heartbeat from `channel: "telegram"` to `channel: "webchat"`
- Webchat is the active working channel — heartbeat now delivers locally to the webchat session
- Gateway restarted
- Queue cleared

**Files created:**
- `SUBAGENT_DELIVERY_GAP_BUG_HUNT.md`
- `HEARTBEAT_TELEGRAM_QUEUE_FIX.md`
