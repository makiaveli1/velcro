# Usability Guide — "Don't Make Me Think" by Steve Krug

A step-by-step guideline for reviewing or creating designs based on Krug's principles. Good design doesn't make users think unnecessarily — everything should be self-evident.

## Step 1: Prepare and Internalize Core Principles

Build a strong foundation before designing.

- **Key ideas**: Users scan (not read), satisfice (pick the first okay option), and muddle through. Prioritize clarity, conventions, and simplicity over innovation.
- **Define scope**: Write down the design's purpose (e.g., "e-commerce site for books"), target users (e.g., "busy readers aged 25-45"), and main goals (e.g., "easy browsing and checkout").
- **Gather references**: Collect examples of good/bad designs from similar sites. Ask: What makes this effortless? What confuses me?

**Output**: A one-page summary of principles and project basics to refer back to.

---

## Step 2: Assess or Plan the Overall Structure

Make the design's hierarchy and purpose obvious at a glance.

- **New design**: Sketch a high-level wireframe. Outline main sections using a simple hierarchy — top-level navigation should be persistent and conventional (menu at top, logo upper-left).
- **Reviewing existing design**: Walk through as a user. Note where the purpose isn't clear or navigation feels hidden. Krug's rule: Clicks are fine if each is unambiguous.
- **Check for self-evidence**: Ensure site identity (logo, tagline) and key actions (e.g., "Shop Now") are prominent. Avoid clever names — use straightforward labels like "Books" instead of "Library Vault."
- **Simulate user paths**: Role-play different scenarios (e.g., "Find a product quickly"). Red-flag anything requiring thought.

**Output**: Wireframes or annotated screenshots with fixes.

---

## Step 3: Optimize for Scanning and Intuitive Interaction

Users glance for cues — design for quick comprehension.

- **Scannable elements**: Use clear headings, short paragraphs, bullet points, and visual hierarchy (bigger/bolder for important items). Highlight keywords.
- **Obvious interactions**: Ensure buttons look clickable (shaded, bordered), links are colored/underlined, and forms are simple (no unnecessary fields).
- **Conventions**: Place search bars top-right, use familiar icons (magnifying glass for search). Minimize noise — remove distracting elements.
- **Billboard rule**: Scan each page in 5 seconds. What stands out? Fix clutter by cutting half the words, then half again.

**Output**: Refined layouts with notes on changes (e.g., "Shortened text by 50% for scannability").

---

## Step 4: Eliminate Unnecessary Cognitive Load

Reduce "noise" and make everything effortless.

- **Simplify content**: Edit ruthlessly — focus on what's essential for user tasks. Avoid jargon; use plain language.
- **Handle edge cases**: Design for errors gracefully (clear "No results" messages with suggestions). Ensure consistency across pages (same button styles).
- **Build iteratively**: Start minimal for new designs. For reviews, identify pain points like confusing instructions or hidden features.
- **Litmus test**: Does this force thinking, or is it intuitive?

**Output**: Cleaned-up prototypes or a list of simplifications.

---

## Step 5: Conduct Solo Usability Testing

Testing is crucial, even informally.

- **Create test scenarios**: Write 3-5 tasks based on user goals (e.g., "Find and buy a mystery novel").
- **Self-test**: Use your prototype or existing design. Record where you hesitate, guess, or backtrack. Note fixes.
- **Krug's metrics**: Fix anything that makes users "think" (unclear labels, hidden actions). Test on mobile too — ensure it adapts without issues.
- **Screen recording**: Capture sessions and review objectively: "Would a new user struggle here?"

**Output**: A testing log with issues and resolutions.

---

## Step 6: Iterate, Refine, and Finalize

Designs improve through cycles, not perfection on first try.

- **Review changes**: Go through steps 2-5 again with updates. Ensure accessibility (alt text, contrast) and mobile-friendliness.
- **Measure success**: Is it self-explanatory? Does it follow conventions? Low noise?
- **Document decisions**: Create a final report summarizing decisions tied back to principles (e.g., "Applied scanning rule by using bullets").
- **Avoid over-iteration**: Set a deadline. Use checklists as templates.

**Output**: Final design files and a self-review doc.

---

## Quick Reference Checklist

Use this during implementation to verify Krug compliance:

- [ ] Page purpose is obvious within 5 seconds
- [ ] Navigation is persistent, conventional, and clearly labeled
- [ ] Site identity (logo, tagline) is visible on every page
- [ ] CTAs are prominent and use action verbs ("Shop Now", not "Click Here")
- [ ] Content is scannable: headings, bullets, highlighted keywords
- [ ] Buttons look clickable; links are distinguishable from text
- [ ] No jargon — plain language throughout
- [ ] Error states are graceful with helpful recovery suggestions
- [ ] Consistent styling across all pages (buttons, colors, spacing)
- [ ] Mobile-responsive with no loss of core functionality
- [ ] Accessibility: alt text, sufficient contrast (WCAG AA), keyboard navigable
